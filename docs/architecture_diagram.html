ThreatByte-MCP: Two Client Modes (Web/HTTP vs Local/stdIO)

This diagram shows (A) the web application acting as an HTTP MCP client via /mcp-proxy, and (B) a local agent/IDE acting as a stdio MCP client by spawning the server as a subprocess.
Interaction types shown: initialize, tools/list, tools/call, streaming responses (SSE in HTTP mode), and side effects (DB writes, file uploads, tool registry changes, LLM calls).

Notes for Demos

  • Web mode is what the ThreatByte UI uses: the browser never talks to the MCP server directly; it calls /mcp-proxy, which forwards JSON-RPC to the MCP HTTP endpoint.
  • stdio mode is intended for IDE/agent clients that spawn the server process and communicate over stdin/stdout. The web app will not work with stdio mode.
  • Streaming: in HTTP mode, agent tools can be streamed back to the browser via SSE; in stdio mode, the client receives incremental events on stdout (client-dependent).
